2. Executive summary
--------------------
Yusra Engineering Ai is architected as a highly portable, privacy-centric, multi-modal conversational assistant designed to ingest text, voice, and images while generating text and TTS audio responses. The system scope requires a mobile-first Progressive Web App (PWA) client that interacts with a modular API gateway, delegating heavy processing to asynchronous worker fleets for ML tasks (OCR, TTS, speaker verification) and RAG operations via a vector store. Development is tightly constrained specifically to execute locally on an Android device via Termux + Proot Ubuntu, mandating scalable but hardware-efficient orchestration that seamlessly transitions to a standard Debian/Ubuntu VPS for production. All system components are strictly governed by RBAC, detailed data retention policies, and the absolute requirement to rigorously enforce the Yusra persona payload.

Component Inventory & Responsibilities:
- Client (PWA/Next.js): Renders mobile-first UI, manages local caching/offline state, handles media capture, and communicates with backend via REST/WebSockets.
- Reverse Proxy (Nginx): Terminates TLS, caches static assets, rate-limits initial connections, and routes requests to the API Gateway.
- API Gateway (Node.js): Handles API routing, user authentication, RBAC authorization, and orchestrates synchronous operations.
- Redis (Cache & Message Broker): Manages session tokens, rate-limiting counters, and BullMQ/RQ job queues for asynchronous and ML-related tasks.
- Worker Fleet: Independently polls Redis queues to process heavy operations (TTS, OCR, document chunking, embeddings, and speaking verification) and writes outcomes to DB/Storage.
- Postgres (Database): Durable relational storage for users, conversation metadata, RBAC rules, RBAC policies, and audit logs.
- Vector Store (pgvector/Milvus): Indexes and queries embeddings for Retrieval-Augmented Generation (RAG) and semantic search over conversation histories.
- Object Storage (MinIO/S3): Persists large unchanging files (user uploads, audio segments, generated documents).
- ML Models: Core AI logic handling classification, summarization, TTS synthesis, embeddings generation, and conversational text-generation.

4. Component inventory (detailed)
---------------------------------
Frontend (PWA)
- Responsibilities: Render mobile-optimized interfaces, capture user media (audio/voice), execute local context management.
- Pages/UX flows: Onboarding (consent & persona intro), Chat/Interaction view, Settings, Media History.
- Offline behavior: Service worker caches static assets/shells. IndexedDB buffers outgoing prompts/audio if offline and syncs when reconnected.
- Security: Secure local storage for tokens, CSP headers, XSS prevention.

Backend (API Gateway)
- Endpoints: RESTful paths prefixed with /api/v1/. E.g., /api/v1/chat, /api/v1/auth.
- Protocols: HTTPS POST/GET, WebSockets/Server-Sent Events (SSE) for streaming model responses.
- Versioning: URI path versioning (/v1/).
- Error model: Standardized JSON ({ "error": { "code": 401, "message": "Unauthorized" } }).

Worker services
- Tasks: Asynchronous LLM processing, TTS audio generation, Video/OCR pipelines, file chunking.
- Queueing: Redis-backed BullMQ streams.
- Idempotency & Failure modes: Job ID deduplication; auto-retry exponential backoff up to 3 times before moving to Dead Letter Queue (DLQ).

Databases and storage
- Schemas summary: User table, Auth credentials, Conversation thread mappings, System logs.
- Retention: Hot data retained for 90 days, cold storage archived via chron.
- Backups: Automated daily snapshots for Postgres via pg_dump.

Vector store and embeddings
- Storage: Milvus or pgvector table storing document chunks and session vectors.
- Indexing: HNSW (Hierarchical Navigable Small World) index for fast approximate nearest neighbor search.
- Retrieval strategy: Cosine similarity matching, re-ranking before injecting into context window.

ML components
- LLM orchestration: Langchain/LlamaIndex routing queries. Main text model: Gemini Pro/Claude 3.5 Sonnet.
- TTS: ElevenLabs or Google Cloud TTS for voice output.
- OCR/Vision: Gemini Pro Vision.
- Speaker verification: Pyannote/Resemblyzer-based embeddings check for identity verification.

Infra
- Reverse proxy & TLS: Nginx with Let's Encrypt managed certificates.
- Monitoring & Logging: Prometheus/Grafana or Datadog for infra. ELK stack or Loki for structured logs.
- Secrets management: Vault/AWS Secrets Manager to inject environment variables securely decoupled from code.

CI/CD
- Pipelines: GitHub Actions triggering lints, unit tests, E2E tests.
- Triggers: Push to main => staging deploy. Tag release => prod deploy.
- Artifact registry: Docker Hub or GCP Artifact Registry.

Observability
- Metrics: Throughput latency per LLM call, endpoint RPS, redis queue depth.
- Logs & Tracing: OpenTelemetry spans injected into API request boundaries. Alerting mapped to queue depth anomalies and 5xx frequency spikes.

Security
- Auth: JWT-based stateless auth for API Gateway, HttpOnly secure cookies for clients.
- RBAC: Scopes bound to tokens: user vs admin endpoints.
- Encryption & Hardening: KMS envelope encryption for sensitive variables at rest.

7. Storage and object lifecycle
-------------------------------

Storage Service Strategy:
The application leverages an S3-compatible object storage service (e.g., AWS S3, Cloudflare R2, MinIO for local dev) to store user-uploaded media (images, audio). The database only stores metadata and references (URLs) to these objects.

Object Key Patterns:
All uploaded objects follow a strict naming hierarchy to ensure isolation and ease of lifecycle management.
- Pattern: `{environment}/{resource_type}/{user_id}/{year}/{month}/{uuid_v4}-{filename}`
- Example: `production/audio/a1b2.../2026/06/f4c5...-recording.webm`
- Example: `production/images/a1b2.../2026/06/e3d4...-photo.jpg`

Signed URL Flow for Uploads and Downloads:
Direct client-to-storage uploads are required to offload bandwidth from the API Gateway.
1. Client requests upload: Client POSTs `/api/v1/media/request-upload` with filename and mime-type.
2. Server validates: API validates the MIME type and file size limits.
3. Server generates Pre-Signed URL: Server generates an upload URL (PUT) valid for 15 minutes and returns it to the client.
4. Client uploads directly: Client PUTs the file directly to the S3 bucket using the signed URL.
5. Client confirms upload: Client POSTs `/api/v1/media/complete-upload` containing the object reference to finalize the DB record.
6. Downloads (Reads): Private objects generate temporary signed GET URLs when requested by authorized clients, valid for 1 hour. Publicly accessible assets (if any) are served via CDN without signed URLs.

Retention and Lifecycle Rules:
Configured directly on the S3-compatible service bucket policies.
- Temporary / Orphaned Uploads: Objects with the prefix `temp/` or not confirmed in the DB are purged after 24 hours via a bucket lifecycle rule.
- Deleted Accounts Media: Objects prefixed with `{environment}/{resource_type}/{user_id}/` where `user_id` matches a deleted account are tagged for deletion and purged after 30 days (soft-delete window).
- Ephemeral Media/Audio: Raw TTS generations or intermediary normalized audio files (if not explicitly pinned) are purged after 7 days.

Backup Schedule:
- The S3 bucket utilizes cross-region replication (CRR) for disaster recovery.
- Object Versioning is ENABLED to protect against accidental overwrites or deletions.
- Regular DB backups refer to these retained object versions.

8. Authentication and authorization
-----------------------------------

Authentication mechanism uses a dual-token strategy: short-lived Access Tokens and long-lived Refresh Tokens.

1. JWT Lifecycle & Cookie vs Header Usage:
- Access Token (JWT): Lifespan 15 minutes, signed using ES256 algorithm. Contains user ID and role claims. Delivered to the client in the login response payload and stored in memory. Sent via the `Authorization: Bearer <token>` HTTP header for all protected API requests.
- Refresh Token: Opaque secure random string (or encrypted JWT). Lifespan 7 days. Stored in an `HttpOnly`, `Secure`, `SameSite=Strict` cookie to mitigate XSS and CSRF risks.
- Token Rotation: The refresh token is rotated upon use. When the client calls the `/api/v1/auth/refresh` endpoint using the HTTP-only cookie, a new access token and a new refresh token are issued. The old refresh token is marked as revoked.

2. OAuth Connectors:
- System supports primary authentication via Google OAuth 2.0.
- Flow: Client redirects to Google consent screen -> Callback to Backend -> Backend exchanges code for profile -> Backend issues local Access + Refresh token pair to the client (same as email/password flow).

3. Two-Factor Authentication (2FA) Flow:
- Method: TOTP (Time-based One-Time Password) via Authenticator apps.
- Setup: User requests 2FA setup, server generates a secret and returns a otpauth URI / QR code. User completes setup by supplying a valid code.
- Login Flow:
  1. User submits email and password.
  2. Server verifies credentials. If 2FA is enabled, returns `{ "requires2FA": true, "tempToken": "ey..." }` instead of login tokens.
  3. User submits TOTP code and `tempToken`.
  4. Server verifies the code, issues standard Access and Refresh tokens.

4. RBAC Matrix with Exact Permissions per Role:
The system employs Role-Based Access Control (RBAC) with two primary roles: `USER` and `ADMIN`.

Role: USER
Permissions:
- `profile:read_own`, `profile:update_own`, `profile:delete_own`
- `conversation:create`, `conversation:read_own`, `conversation:update_own`, `conversation:delete_own`
- `message:create`, `message:read_own`
- `media:upload`, `media:read_own`
- `speaker_profile:enroll_own`, `speaker_profile:read_own`

Role: ADMIN
Permissions:
- All USER permissions (applied system-wide).
- `user:read_all`, `user:update_all`, `user:delete_all`
- `system:read_settings`, `system:update_settings`
- `audit:read_logs`
- `metrics:view_worker_stats`

10. RAG and retrieval design
----------------------------

Retrieval-Augmented Generation (RAG) is core to enabling YUSRA to retrieve contextual engineering documentation, dynamic guidelines, and personalized domain knowledge.

1. Chunking Strategy:
- Document Loader: Parses PDF, TXT, DOCX, and Markdown source documents.
- Splitter: Recursive Character Text Splitter (by paragraphs first, falling back to sentences/words).
- Chunk Size: Primary chunks are sized at 512 tokens.
- Chunk Overlap: An overlap of 64 tokens is configured to maintain linguistic and semantic context across chunk borders.
- Parent-Child Mapping: The system implements hierarchal chunks. Small chunks of 256 tokens are generated and indexed into pgvector for similarity searches, but map back to parent blocks of 1024 tokens which are retrieved and fed into the LLM context to preserve maximum surrounding context.

2. Embedding Model Choice:
- Model: OpenAI `text-embedding-3-small` (or equivalent multi-lingual embedding model) configured for exactly 1536-dimensional vectors. This ensures a 100% matches with the `VECTOR(1536)` dimension in the PostgreSQL `embeddings` table.
- Normalization: Vectors are normalized on the server-side before execution block searches to allow cosine distance query matches to be fast.

3. Similarity Search & Threshold Rules:
- Vector Index Operator: Cosine distance (`<=>` operator in PostgreSQL pgvector).
- Dynamic Distance Threshold: Matching chunks must have a cosine similarity threshold of >= 0.72. Chunks below this threshold are immediately filtered out.
- Initial Retained Set (K-value): Fetch top K=25 candidate chunks from the index.

4. Re-ranking Rules:
- Engine: A local or API-driven Cohere Multilingual Re-ranker (`rerank-multilingual-v3.0`).
- Process: The initial top K=25 chunks retrieved from pgvector are passed to the re-ranking service along with the user's explicit query.
- Final Retained Pool: The top K=5 chunks with a re-ranked relevance score >= 0.65 are passed to the prompt assembler.
- Fallback: If the re-ranker is down or returns errors, the system falls back to the top 5 raw cosine similarity chunks from pgvector.

5. Provenance Metadata Contract:
Every retrieved chunk must carry the following structured provenance metadata:
- `document_id`: UUID reference to parent source resource.
- `source_title`: String descriptive name of source file.
- `page_number`: Integer reference (if source was a multi-page document).
- `chunk_index`: Integer positional tracking index.
- `last_updated`: Timestamp of parent asset modification.

When YUSRA responds using retrieved documents, the LLM is explicitly instructed to cite sources using markdown footnotes linked to these metadata keys.

11. Worker queue and task design
--------------------------------

The asynchronous execution of resource-intensive tasks (such as media transcoding, document chunking/embeddings, and notifications) is managed by background workers orchestrated using a message queue system (e.g., BullMQ with Redis or RabbitMQ).

1. Queue Names and Roles:
- `media-transcoding`: Handles audio/video conversion and normalization tasks.
- `embedding-indexing`: Handles document loading, text chunking, embedding generation via OpenAI/Gemini API, and storage in pgvector.
- `notification-outbound`: Dispatcher for push notifications, transactional emails, and system updates.

2. Dynamic Message Flow Architecture:
- Client uploads raw media object -> S3 lifecycle triggers database metadata update -> Web Server publishes a `TRANSCODE` event to `media-transcoding` queue.
- Background worker picks up task -> нормализует/converts file format using FFmpeg -> saves the canonical asset into S3 -> marks message status as ready -> triggers an `INDEX` event to `embedding-indexing` queue if content contains transcribable text.

3. Payload Schemas:

- Task: `media-transcoding.process`
  ```json
  {
    "taskId": "task_uuid_v4",
    "resourceId": "media_uuid_v4",
    "sourceUrl": "https://s3.example.com/raw/audio.webm",
    "targetFormat": "webm-audio-canonical",
    "userId": "user_uuid_v4"
  }
  ```

- Task: `embedding-indexing.generate`
  ```json
  {
    "taskId": "task_uuid_v4",
    "referenceId": "msg_or_doc_uuid_v4",
    "referenceType": "MESSAGE",
    "text": "The content string to chunk and embed.",
    "userId": "user_uuid_v4"
  }
  ```

4. Idempotency Key Handling:
- Every job submitted to a queue must specify a unique `idempotencyKey` inside the job options.
- The `idempotencyKey` is derived deterministically (e.g., `hash(resourceId + taskType)`).
- Before storing/running a task, the queue manager checks if a job with that key is currently active, completed, or failed in Redis:
  - If completed: Returns the output directly without re-running.
  - If active: Deduplicates by chaining the request to the active job's promise/completion event.

5. Retry, Backoff Policy, and Dead-Letter Handling (DLQ):
- Max Retries: 5 attempts.
- Backoff Type: Exponential.
- Backoff Parameters: Initial delay = 5000ms (5 seconds), exponent factor = 2.0 (i.e. retry at 5s, 10s, 20s, 40s, 80s).
- Dead-Letter Queue (DLQ): 
  - If a job fails after 5 attempts, it is moved to a dead-letter state (e.g., `media-transcoding-dlq`).
  - The failure context, stack trace, and final attempts metadata are persisted globally.
  - An alert is fired when the DLQ size exceeds a critical threshold.
  - Workers do not execute DLQ tasks automatically. Administrators must inspect DLQs via a dashboard and select either "Re-queue Task" or "Purge Task".

6. Monitoring Metrics:
- `queue_size`: Number of currently queued/delayed/waiting messages. (Alert trigger: set to > 500 for over 5 mins).
- `processing_latency`: Elapsed time in ms from message pickup to completion. (Target: < 2000ms for transcoding, < 500ms for embeddings).
- `error_rate`: Ratio of failed jobs to successful jobs per minute. (Alert trigger: error rate > 5.0%).
- `dlq_count`: Number of absolute failures parked in the DLQ. (Alert trigger: count > 10).

12. Media processing pipeline
-----------------------------

The media processing pipeline handles transcoding incoming raw audio/video files, normalization, Text-to-Speech (TTS) synthesis orchestration, voice template management, and CDN cache policies.

1. Audio Normalization & Canonical Formats:
To ensure consistent volume across multiple user interactions and different recording devices, all audio assets are normalized using standard EBU R128 guidelines.
- Target Integrated Loudness: -16 LUFS (Loudness Units Full Scale) for speech and conversational audio.
- Max True Peak: -1.0 dBTP (Decibels True Peak) to prevent clipping.
- Canonical Formats:
  - Streaming Audio: WebM container with Opus codec, 48kHz sampling rate, mono, 64kbps bit rate. Highly optimized for mobile devices and modern browser decoders.
  - Legacy Fallback: MP3 container, 44.1kHz sampling rate, stereo, 128kbps bit rate.
  - Video Assets: MP4 container with H.264 video codec (yuv420p, high profile) and AAC-LC stereo audio codec.

2. FFmpeg Transcoding Presets:
Transcoding operations run within the background `media-transcoding` worker namespace. The worker executes FFmpeg via child processes with these hardened production presets:

- Command 1: Loudness Normalization (Two-Pass loudnorm filter)
  Pass 1 (Analyze):
  `ffmpeg -i input_raw.wav -af loudnorm=I=-16:TP=-1.0:LRA=11:print_format=json -f null -`
  Pass 2 (Apply using analyzed values):
  `ffmpeg -i input_raw.wav -af loudnorm=I=-16:TP=-1.0:LRA=11:measured_I={measured_i}:measured_TP={measured_tp}:measured_LRA={measured_lra}:measured_thresh={measured_thresh}:offset={measured_offset}:linear=true -acodec libopus -b:a 64k -ar 48000 output_canonical.webm`

- Command 2: Legacy MP3 Transcode Fallback
  `ffmpeg -i input_raw.wav -vn -acodec libmp3lame -q:a 2 -ar 44100 output_fallback.mp3`

3. TTS Voice Management & Synthesis:
The YUSRA application integrates with advanced TTS providers (e.g., ElevenLabs or Google Cloud Text-to-Speech) to generate human-like emotional audio responses.
- Engine: Google Cloud TTS (Neural2/Wavenet models) or ElevenLabs Multilingual v2.
- Dynamic Voices: Pre-enrolled tone profiles represent YUSRA's custom voices (friendly, educational, soft-toned).
- TTS Caching Policy:
  - Cache Key: Generated deterministically on the backend API Gateway using SHA-256 of the input data: `SHA256(voice_id + "_" + raw_text_content + "_" + model_settings_hash)`.
  - Cache Storage:
    - Hot Storage (Redis): Stores a reference metadata object mapped to the asset path.
    - Cold Storage (S3 Bucket): The synthesised `.mp3` or `.webm` output is saved in the bucket under `production/media/tts_cache/{cache_key}.webm`.
  - Retention: Cache hits reset the lease. Unaccessed cached TTS voice files are deleted after 14 days to minimize object storage costs.

4. CDN Delivery and Caching Strategies:
Media files are highly static once transcoded and normalized, requiring a robust caching architecture at the edge.
- CDN Architecture: Cloudflare or Google Cloud CDN endpoints front the media S3 bucket.
- Header Configuration:
  - Canonical Transcoded Audio: `Cache-Control: public, max-age=31536000, immutable` (permanently cached on CDN and browser).
  - Dynamic TTS Cache: `Cache-Control: public, max-age=1209600, stale-while-revalidate=86400` (allows edge caching for 14 days, with 1-day graceful revalidation).
- Origin Shielding: CDN Edge rules rewrite access requests so that direct requests to private buckets are offloaded through signed CDNs or edge worker authentication hooks.

13. Speaker verification enrollment and verification flows
----------------------------------------------------------

Speaker Verification dynamically validates and authenticates a user's voice against their biometrically enrolled identity profile.

1. Enrollment UX & Guidelines:
The enrollment process is design-optimized for clear biometric collection:
- Step 1: Pre-enrollment Quality Check:
  - User is prompted to find a quiet environment. 
  - System captures 3 seconds of background noise. If background decibels (dB) exceed -45dB, user is warned of potential high ambient noise.
- Step 2: Guided Audio Recording:
  - User must read 3 distinct phonetically balanced prompt sentences sequentially to record their baseline audio characteristics.
  - Sentence A: "My voice is my unique biometric signature, confirming my identity securely to YUSRA."
  - Sentence B: "Engineering solutions require precision, clarity, and adaptive, intelligent architectures."
  - Sentence C: "YUSRA, I'm enrolling my voice print to safeguard my conversational domain."
- Step 3: Recording Phase & Visual Feedback:
  - The UI presents a clear Waveform Visualizer displaying real-time audio amplitude variations.
  - A recording timer tracks progress. Each utterance must possess a duration between 3.0 and 6.0 seconds.
- Step 4: Verification Gate (Quality Control):
  - High and low pass frequency filters analyze the recorded voice.
  - If clipping (signal peaks >= 0 dB) or sub-standard signal-to-noise ratio (SNR < 15 dB) is detected, the trial is rejected and user is prompted to re-record.

2. Speaker Embedding Storage Format:
- Embedding Model: An advanced d-vector extraction network (e.g., ECAPA-TDNN trained on VoxCeleb, or fine-tuned ResNet-based speaker identification model).
- Dimensionality: Produces a 1536-dimensional float vector, fully matching the `VECTOR(1536)` database definition.
- Storage Target:
  - After a user completes all three sentences, the server averages the three distinct voice d-vectors to construct a unified speaker profile template vector.
  - This averaged template is written to the `embeddings` table:
    - `id`: Generate UUID_v4
    - `user_id`: Reference `users.id`
    - `reference_id`: Reference the corresponding user profile record
    - `reference_type`: String value set exactly to `'SPEAKER_PROFILE'`
    - `content_text`: Store serialized metadata such as `"Voice profile enrolled via Web-PWA"`
    - `embedding`: Store the 1536-dimensional float vector.

3. Matching Metric & Similarity Thresholds:
- Metric: Cosine similarity via the pgvector operator `<=>` (cosine distance).
- Verification Logic:
  - When a user performs voice verification, the backend generates an embedding for the active verification audio segment.
  - It runs a database query to compute the cosine distance against the stored template vector where `reference_type = 'SPEAKER_PROFILE'`.
  - Calculation: `similarity = 1.0 - distance_score`.
- Security Threshold Boundaries:
  - High Confidence Verification: A matches similarity score `>= 0.85` is required to authorize critical actions without extra prompts.
  - Low Confidence or Mismatches: If similarity score falls between `0.70` and `0.84`, the attempt is flagged as "indubitable but low quality", requiring additional credentials.
  - Rejection: Any match rating under `0.70` is immediately rejected.

4. Fallback Verification & Authentication Flows:
- To prevent user exclusion due to temporary voice degradation (e.g., hoarseness, cold, laryngitis) or heavy environmental noise, robust automatic back-up strategies are configured:
  - If voice verification fails or scores under `0.85`, YUSRA prompts the client with: `"We couldn't verify your voice signature. Please try again or authenticate using a secondary method."`
  - Fallback option 1: Secure 6-digit backup Personal Identification Number (PIN) or account password.
  - Fallback option 2: Out-Of-Band (OOB) TOTP authentication via registered Authenticator applications or SMS/Email verification codes.
  - Fallback option 3: Strict Manual review fallback where admins can view audio capture audit metrics.

5. Biometric Consent UI Text:
Before engaging in speaker profiling, the client must check a mandatory authorization option displaying this verbatim policy text:
"""

By proceeding, you explicitly authorize YUSRA to capture, process, and analyze your voice recordings.
